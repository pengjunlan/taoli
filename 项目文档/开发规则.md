# 多交易所套利系统开发规则

## 1. 文档目的

本文件用于统一整理当前项目已经明确的开发规则，作为后续补功能、改页面、接真实后端时的共同基线。

本文件以当前仓库中的真实目录结构、路由实现、认证逻辑和现有页面约定为准，不再沿用已经取消的早期原型思路。


## 2. 当前开发总原则

1. 系统继续围绕四条主线推进：
   - 套利机会发现
   - 自动规则执行
   - 账户资金调度
   - 风控与结果追踪

2. 当前项目默认按多用户系统设计。
   - 后续所有业务数据都必须按用户隔离
   - 不能按“全局共用一套账户、规则、持仓、风控数据”的思路继续开发

3. 先保证链路清晰，再逐步接真实业务。
   - 当前允许页面原型数据和占位数据存在
   - 但目录分层、职责边界、认证链路要按正式系统方式建设

4. 已确认的页面结构、文案、视觉方向，不得在做功能时顺手改动。
   - 如果需要改布局、改模块、改文案、改视觉，需要单独确认

5. 已经取消的无效模块不回加，除非后续有真实业务支撑。


## 3. 当前技术栈与目录规则

### 3.1 技术栈

- 后端框架：FastAPI
- 页面模板：Jinja2
- 前端交互：HTML + CSS + 原生 JavaScript ES Modules
- 当前启动方式：`start_web.bat` 或 `uvicorn`

### 3.2 分层约定

- `app/controller`
  只负责路由、参数接收、调用服务、返回结果

- `app/views`
  只负责展示层，包括模板、静态资源、页面展示模型、页面/API 输出组装

- `app/application`
  负责业务编排和应用服务，例如认证、监控、机会扫描、交易决策

- `app/domain`
  负责核心业务对象、规则、策略约束、领域模型

- `app/infrastructure`
  负责交易所通讯、缓存、持久化、后台任务等基础设施能力

- `app/shared`
  只放真正跨层复用的工具、异常、类型

- `app/core`
  当前只保留少量底层辅助，不再作为通用杂物目录继续扩展

### 3.3 过渡目录规则

以下目录当前仅作为兼容过渡保留，后续新代码不再继续放入：

- `app/communication`
- `app/model`
- `app/service`
- `app/utils`

### 3.4 命名保留规则

- 展示层统一继续使用 `views` 命名，不删除、不改名
- 文件命名继续使用当前仓库已经采用的 `snake_case`


## 4. 页面开发规则

### 4.1 一页一套文件

每个业务页面应尽量保持独立文件对应关系：

- `app/controller/pages/xxx_controller.py`
- `app/views/templates/pages/xxx.html`
- `app/views/static/css/pages/xxx.css`
- `app/views/static/js/pages/xxx.js`

页面控制器、模板、样式、脚本文件名应保持同名语义，便于后续维护。

### 4.2 页面路由汇总规则

- 页面总路由统一汇总到 `app/controller/page_controller.py`
- 每个页面使用独立控制器实现，不把所有页面逻辑堆进总路由文件

### 4.3 页面渲染规则

- 页面控制器通过 `PageConfig` 定义页面元信息
- 页面输出统一走 `app/views/presenters/page_presenters/template_renderer.py`
- 导航、标题、页面壳层等通用信息由渲染器统一注入

### 4.4 页面展示数据规则

- 页面展示模型优先进入 `app/views/viewmodels`
- 当前原型页面上下文统一放在 `app/views/viewmodels/pages/page_contexts.py`
- 页面/API 的展示拼装逻辑优先进入 `app/views/presenters`
- 不再把页面展示数据继续放回旧的 `core` 或过渡目录

### 4.5 页面交互边界

- 页面 JS 只负责展示层交互和接口调用入口
- 交易判断、监控判断、规则判断、执行决策，不写在页面 JS 中

### 4.6 表单标注规则

- 只要是表单中的输入项，都必须明确标注“必填”或“选填”
- 适用范围包括：
  - `input`
  - `select`
  - `textarea`
- 推荐直接写在字段标签中，例如：
  - `账号（必填）`
  - `API Passphrase（选填）`
  - `备注（选填）`
- 如果字段标签会被前端脚本动态切换，切换后的文案也必须保留“必填”或“选填”标识


## 5. 接口与实时通道开发规则

### 5.1 API 汇总规则

- API 总路由统一汇总在 `app/controller/api_controller.py`
- 各模块接口继续拆分在 `app/controller/api/` 目录下

### 5.2 API 职责边界

- `controller/api` 负责接收请求和返回响应
- 展示型响应组装优先放在 `app/views/presenters/api_presenters`
- 业务编排优先放在 `application/services` 或后续 `use_cases`

### 5.3 WebSocket 规则

- 实时通道统一由 `app/controller/api/websocket_api.py` 管理
- 新增实时频道时，继续沿用“频道名 + payload builder”的组织方式

### 5.4 前端调用规则

- 当前前端异步请求统一沿用 `app/views/static/js/core/utils.js` 中的公共方法
- 公共提示、异步请求、退出登录等共用交互优先复用公共工具，不重复散落实现


## 6. 认证与多用户隔离规则

### 6.1 页面访问规则

- 未登录用户访问业务页面时，应跳转到 `/login`
- 登录页、注册页需要支持“已登录则跳转业务首页”的处理

### 6.2 API 访问规则

- 除注册、登录等公开接口外，业务接口默认需要登录校验
- 未登录访问业务接口时，应返回 `401`

### 6.3 WebSocket 访问规则

- WebSocket 通道默认需要登录校验
- 未通过校验时，连接应直接关闭

### 6.4 当前认证实现基线

- 当前认证存储方案为 `MySQL + Redis`
- `MySQL` 保存用户、会话持久化记录、登录日志
- `Redis` 用于会话缓存和快速校验
- 当 `Redis` 不可用时，允许回退到 `MySQL` 做会话校验，保证登录链路可用

### 6.5 多用户隔离硬规则

后续所有真实业务表、查询、写入、执行任务都必须带用户归属，至少包括以下数据：

- 交易所账户
- 自动规则组
- 套利机会结果
- 持仓与订单
- 调拨记录
- 接收地址配置
- 风控告警与风险事件

落地时默认要求：

- 数据表带 `user_id`
- 查询按当前登录用户过滤
- 写入记录用户归属
- 后台任务和执行链路保留用户上下文


## 7. 当前页面与产品一致性规则

### 7.1 页面风格基线

- 当前页面采用浅色数据面板风格
- 以卡片和表格作为核心信息载体
- 顶部导航已固定，不采用左侧复杂菜单
- 页面头部尽量简洁，不堆无实际功能价值的入口

### 7.2 首页规则

- 首页只承担概览和预览职责
- 首页不承担完整列表页职责
- 首页机会列表只展示少量预览，并通过“更多”跳转完整页

### 7.3 机会页规则

- 资金费套利页与价差套利页保持统一表格结构
- 差异主要通过字段含义体现，不做两套完全不同布局
- 资金费页保留“距离结算”
- 价差页保留“建议仓位”

### 7.4 规则页规则

- 规则页展示的是全局规则组
- 不是单个交易对手工任务列表
- 页面要强调“统一扫描、命中规则后执行”的逻辑

### 7.5 账户页规则

- 账户页属于核心业务页，不是单纯 API 录入页
- 需要围绕资金分配、均衡、调拨、地址配置来组织

### 7.6 风控页规则

- 风控页不只是阈值展示
- 需要能解释风险事件如何影响执行和收益稳定性

### 7.7 当前不再保留的内容

以下内容当前不再作为默认设计方向：

- 观察按钮
- 导出按钮
- 大量筛选区和搜索区
- 顶部搜索框
- 顶部用户中心按钮
- 首页收益曲线模块
- 演示 / 原型 / 开发中文案
- 围绕单个交易对手工创建套利任务的设计思路


## 8. 新增功能时的执行规则

### 8.1 新增页面时

建议按以下顺序落地：

1. 新建页面控制器，并定义 `PageConfig`
2. 新建对应模板、页面 CSS、页面 JS
3. 在 `page_controller.py` 中汇总路由
4. 如需进入主导航，在 `views/viewmodels/navigation.py` 中补充导航项
5. 页面需要登录保护时，接入当前认证判断
6. 页面展示数据优先放入 `views/viewmodels` 或后续拆分模块

### 8.2 新增 API 时

建议按以下顺序落地：

1. 在 `app/controller/api/` 下新增模块
2. 在 `api_controller.py` 中汇总路由
3. 默认接入登录校验
4. 业务逻辑放到 `application`
5. 输出组装放到 `views/presenters` 或 DTO
6. 所有真实数据读写都带用户归属

### 8.3 新增真实业务能力时

- 先保持页面结构稳定
- 再接真实数据
- 再接真实执行逻辑
- 再接真实配置能力
- 最后再决定是否需要增加更复杂交互


## 9. 当前结论

当前项目已经从“页面原型堆叠”进入“结构已确定、分层已展开、认证已接入”的阶段。

后续开发应继续遵守以下基线：

- 不破坏当前分层
- 不绕开认证与用户隔离
- 不把业务逻辑写回页面层
- 不擅自改动已确认页面
- 围绕主链路持续把原型能力替换为真实能力
