多交易所套利系统文件结构说明

一、当前技术实现

1. 后端框架：FastAPI
2. 页面模板：Jinja2
3. 前端交互：HTML + CSS + 原生 JavaScript ES Modules
4. 当前数据形态：页面原型数据 / 占位数据
5. 当前启动方式：`start_web.bat` 或 `uvicorn`

说明：
1. 当前项目已经开始从页面原型结构，过渡到分层更清晰的正式结构
2. 展示层继续保留 `views` 命名，不删除、不改名
3. 后续真实接入时，重点围绕账户监控、市场监控、机会分析、开平仓决策逐步落地
4. 后续接入真实后端时，所有核心数据都需要带用户归属字段

三、当前页面文件存放规范

1. 每个页面单独一套文件
   `views/templates/pages/xxx.html`
   `views/static/css/pages/xxx.css`
   `views/static/js/pages/xxx.js`

2. 每个页面都应该有自己的页面控制器
   `controller/pages/xxx_controller.py`

3. 页面总路由统一保留在 `controller/page_controller.py`
   它负责汇总页面控制器，不承载所有页面细节实现

4. API 总路由统一保留在 `controller/api_controller.py`
   后续新增接口、WebSocket、推送通道，优先从这里统一汇总

5. 页面展示数据优先进入 `views/viewmodels`
   不再继续放入旧的 `core`

6. 页面渲染和 API 输出封装优先进入 `views/presenters`

7. 交易、监控、分析、执行逻辑不写在页面 js 中
   页面层只负责展示与交互入口


四、当前各层职责备注

1. `controller`
   只负责路由、参数接收、调用服务、返回结果

2. `views`
   只负责展示层，包括模板、静态资源、页面展示模型、页面/API 输出组装

3. `application`
   负责业务编排，如账户监控、市场监控、机会扫描、开平仓决策

4. `domain`
   负责核心业务对象、规则、策略约束

5. `infrastructure`
   负责交易所通讯、市场数据连接、缓存、持久化、后台任务

6. `shared`
   负责真正通用的工具、异常、类型

7. `core`
   当前只保留极少数底层辅助，不再作为“什么都能放”的目录


五、当前结构设计原则

1. 保留 `views` 作为展示层，不删除、不改名
2. 页面结构、样式、交互分离
3. 页面路由集中汇总，但每个页面单独控制器实现
4. API 路由集中汇总，后续再按模块拆分到 `controller/api`
5. 展示模型、领域模型、业务服务、基础设施分层隔离
6. 通讯层不直接承担业务判断
7. 工具层只放真正可复用的跨层工具
8. 先保证页面链路清晰，再逐步接真实业务逻辑
9. 围绕“机会发现、规则执行、资金调度、风控追踪”四条主线组织系统
10. 后续所有账户、规则、持仓、订单、风控数据都必须按用户隔离设计
