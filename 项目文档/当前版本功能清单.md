# 多交易所套利系统当前版本功能清单

## 1. 文档说明

本清单以当前项目中已经实现的页面、路由、文案和交互结构为准。

当前版本不再按照“单个交易对手工建策略”的思路设计，而是围绕以下主链路展开：

- 已配置交易所统一纳入监控范围
- 系统自动扫描资金费套利机会
- 系统自动扫描跨交易所价差套利机会
- 命中统一规则组后按参数直接执行
- 账户调度、资金均衡、地址配置作为基础能力
- 持仓、订单、风控和风险事件围绕自动套利执行链路展开
- 系统需要支持多用户同时使用，且用户之间数据完全隔离


## 2. 当前页面总览

当前版本共 9 个页面：

- 登录页 `/login`
- 注册页 `/register`
- 首页 `/dashboard`
- 资金费套利页 `/funding-arbitrage`
- 价差套利页 `/spread-arbitrage`
- 自动规则管理页 `/strategies`
- 持仓与订单页 `/positions-orders`
- 账户与资金调度页 `/accounts`
- 风控与告警页 `/risk-alerts`


## 3. 当前产品主逻辑

### 3.1 机会统一扫描

- 系统不是按单个交易对手工创建“是否套利”
- 只要交易所账户已经配置并纳入系统，就进入统一监控范围
- 系统统一扫描两类机会：
- 资金费套利
- 跨交易所价差套利
- 候选机会命中规则后直接进入执行队列

### 3.2 规则组统一驱动

- 规则是全局规则组，不是某个币种的单独开关
- 规则组按套利类型和覆盖范围划分
- 规则组决定：
- 扫描范围
- 收益门槛
- 执行条件
- 仓位规模
- 风险限制

### 3.3 资金调度是主能力

- 账户页不是单纯账户录入页
- 它同时承担资金分配、自动均衡、调拨跟踪、地址配置等职责
- 当资金失衡时，需要支持：
- 一键平均分配
- 自动失衡修复
- 调拨任务跟踪
- 接收地址配置

### 3.4 风控贯穿执行链路

- 风控影响自动开仓
- 风控影响资金调度
- 风控影响持仓完整度
- 风控事件需要能回溯到策略、账户、连接、订单等对象

### 3.5 多用户隔离

- 系统必须支持多个用户同时使用
- 每个用户的数据彼此独立
- 一个用户不能看到另一个用户的任何业务数据
- 以下数据都必须按用户隔离：
- 交易所账户
- 自动规则组
- 套利机会结果
- 持仓与订单
- 调拨记录
- 接收地址配置
- 风控告警与风险事件
- 后续接入真实后端时，所有查询、写入、执行任务都必须带用户归属

- 类型

## 6. 当前版本需要保持一致的设计原则

- 首页只负责预览，不承担完整列表职责
- 资金费套利和价差套利必须明确区分
- 所有主列表页面尽量保持统一表格结构
- 不保留“观察”相关无效按钮
- 不保留“导出”按钮
- 不保留演示、原型、开发中说明文案
- 页头尽量收敛，不堆无实际价值操作
- 表格中优先展示业务信息，不优先展示操作按钮
- 做功能时不得擅自修改已经确认过的页面
- 如果功能接入需要改页面结构、布局、文案或视觉表现，必须先单独确认

## 8. 当前版本已新增的基础认证能力

- 已实现账号注册，后端会校验账号格式、密码长度和确认密码一致性
- 已实现账号登录，页面和后台通过 AJAX 异步通信
- 登录失败时，前端会给出失败提示
- 注册成功后跳转登录页，登录成功后跳转首页
- 已实现退出登录，顶部退出登录按钮会异步调用后端接口
- 未登录用户不能访问业务页面，访问时会被重定向到 `/login`
- 未登录用户不能访问业务数据接口，接口会返回 `401`
- WebSocket 实时通道也增加了登录校验
- 系统按多用户方式设计，后续所有业务表都必须带 `user_id` 做数据隔离
- 当前认证存储方案为 `MySQL + Redis`
- `MySQL` 保存用户、登录日志、持久化会话记录
- `Redis` 用于会话缓存与快速校验
- 当 Redis 暂时不可用时，系统会自动回退到 MySQL 会话校验，保证登录链路可用


## 9. 当前结论

当前版本页面已经稳定围绕四条主线组织：

- 套利机会发现
- 自动规则执行
- 账户资金调度
- 风控与结果追踪

后续继续补功能时，建议继续围绕这四条主线推进，不再回到“传统后台配置页优先”的思路。
